# Bookish — The Everyday Store

A school-mapped, **bundle-driven e-commerce platform** built with **Laravel**, implementing the full `shopbookish.com` specification: School → Class → Bundle selling, guest checkout, passwordless OTP login (WhatsApp + SMS fallback), Direct Bank Transfer / Raast payments with manual proof verification, an 11-module admin panel, RBAC, and SEO routing.

> **Important — read this first.** This download is the **custom application code** (models, migrations, controllers, services, routes, Blade views, seeders). It is **not** a full installed Laravel app — it does not include the `vendor/` folder or the framework skeleton, because those must be generated by Composer on your own machine. The steps below create a fresh Laravel project and drop these files into it. It takes about 10 minutes.

---

## 1. Requirements

Install these on your machine first:

- **PHP 8.2 or newer** (with extensions: `mbstring`, `xml`, `ctype`, `json`, `bcmath`, `fileinfo`, `pdo_mysql`)
- **Composer** (https://getcomposer.org)
- **MySQL 8** (or MariaDB 10.4+)
- **Node.js 18+** and **npm** (only if you later want to compile assets; the views use the Tailwind CDN, so this is optional to get running)
- **Git** (https://git-scm.com)

On Windows the simplest path is [Laragon](https://laragon.org) or [XAMPP](https://www.apachefriends.org) (both bundle PHP + MySQL). On macOS use [Herd](https://herd.laravel.com). On Linux install `php`, `composer`, and `mysql-server` via your package manager.

---

## 2. Create a fresh Laravel project

In a folder of your choice:

```bash
composer create-project laravel/laravel bookish
cd bookish
```

This downloads Laravel and the `vendor/` folder. You now have a clean Laravel app.

---

## 3. Copy these files into the project

Copy **everything from this download** into the `bookish` folder you just created, **overwriting** when prompted. The folder layout already matches Laravel, so files land in the right place:

- `app/` → models, controllers, middleware, services, form requests
- `database/migrations/` and `database/seeders/`
- `routes/web.php`, `routes/api.php`, `routes/admin-auth.php`
- `resources/views/`
- `.gitignore`

When asked to overwrite `app/Models/User.php`, the default migration `0001_01_01_000000_create_users_table.php`, `routes/web.php`, and `database/seeders/DatabaseSeeder.php` — **say yes**. These are intentionally replaced.

> The `_docs/` and `config/services_SNIPPET.php` files are **instructions, not drop-in files**. You'll merge them by hand in step 5–6. You can delete the `_docs/` folder afterward.

---

## 4. Install Sanctum (API tokens)

```bash
composer require laravel/sanctum
```

The API auth controller issues Sanctum tokens, so this package is required.

---

## 5. Wire up routing & the `role` middleware

Open **`bootstrap/app.php`** and edit the `->withRouting(...)` and `->withMiddleware(...)` calls so the API routes load and the `role` alias is registered. Use `_docs/bootstrap-app.php.txt` as the reference — it should look like this:

```php
->withRouting(
    web: __DIR__.'/../routes/web.php',
    api: __DIR__.'/../routes/api.php',
    commands: __DIR__.'/../routes/console.php',
    health: '/up',
)
->withMiddleware(function (Middleware $middleware) {
    $middleware->alias([
        'role' => \App\Http\Middleware\EnsureUserRole::class,
    ]);
})
```

(If `routes/api.php` doesn't exist after `create-project`, the copy in step 3 provides it; the line above tells Laravel to load it.)

---

## 6. Add the OTP service config

Open **`config/services.php`** and merge in the keys from `config/services_SNIPPET.php` (the `whatsapp` and `sms` arrays). They read from the `.env` variables added in the next step.

---

## 7. Configure your environment (`.env`)

Open `.env` and set the values shown in `_docs/env-additions.txt`. At minimum:

```env
APP_NAME=Bookish
APP_URL=http://127.0.0.1:8000

DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=bookish
DB_USERNAME=root
DB_PASSWORD=

FILESYSTEM_DISK=public

# Fill these when you have live credentials.
# Leave blank during local testing — OTP codes are returned
# in the response while APP_ENV=local so you can still log in.
WHATSAPP_PHONE_ID=
WHATSAPP_TOKEN=
SMS_GATEWAY_ENDPOINT=
SMS_GATEWAY_API_KEY=
SMS_GATEWAY_SENDER=Bookish
```

Create the MySQL database named `bookish` (e.g. in phpMyAdmin, or `mysql -u root -e "CREATE DATABASE bookish;"`).

Then generate the app key:

```bash
php artisan key:generate
```

---

## 8. Migrate, seed, and link storage

```bash
php artisan migrate --seed
php artisan storage:link
```

- `--seed` loads categories (Books / Uniforms / Accessories + sub-categories), the three schools (PAFWA Education System, PAF Montessori School, Fazaia School System) each with Classes 1–5, default Meezan bank details, and the super-admin account.
- `storage:link` makes uploaded payment proofs and product images publicly viewable.

---

## 9. Run it

```bash
php artisan serve
```

Open **http://127.0.0.1:8000**.

| Area | URL |
|------|-----|
| Storefront | `/` |
| Browse schools / bundles | `/schools` |
| Customer OTP login | `/login` |
| Admin panel login | `/admin/login` |

**Default super-admin login** (change immediately):

- Mobile: `03000000000`
- Password: `password`

> Customers log in with OTP (no password). Admins/super-admins log in with mobile + password at `/admin/login`. While `APP_ENV=local`, the OTP code is returned in the API/login response so you can test without live WhatsApp/SMS credentials.

## 10. Where the spec features live

- **Bundle pricing** (auto total → % discount → final): `app/Services/BundlePricingService.php`, admin UI at `/admin/bundles/create` (School → Class → Add Books → auto-calculate).
- **Toggle individual books in a bundle**: storefront `resources/views/storefront/bundle.blade.php` (per-book checkboxes build `exclude[]`), handled in `Storefront/CartController::addBundle`.
- **Publisher masking**: `publisher` is in `$hidden` on `app/Models/Product.php`; admin views call `makeVisible('publisher')`. Never exposed on public pages or the public API.
- **OTP (WhatsApp primary, SMS fallback)**: `app/Services/Otp/` — `OtpService` tries WhatsApp, falls back to SMS, stores a hashed 6-digit code with 5-minute expiry and rate limiting.
- **Payment proof workflow**: checkout → bank details page → upload screenshot → admin split-screen verify at `/admin/payments` → approve flips order to paid/processing.
- **RBAC**: `app/Http/Middleware/EnsureUserRole.php`, applied as `role:admin,super_admin`. Finance, Admin-Users, and Settings modules are super-admin only.
- **SEO**: routes `/school/{school}/{class}/bundle`, `/product/{product}`, `/category/{type}`; dynamic meta/OpenGraph in `resources/views/layouts/app.blade.php`; JSON-LD on product pages.
- **API contracts**: `routes/api.php` matches the spec (send-otp / verify-otp throttled, products, bundles, orders, admin-protected CRUD).

## 11. Attach Git and push to a repository

### 11.1 Initialize Git locally

From inside the `bookish` project folder:

```bash
git init
git add .
git commit -m "Initial commit: Bookish e-commerce platform"
```

The included `.gitignore` already excludes `vendor/`, `.env`, and uploaded files — secrets and dependencies will **not** be committed, which is correct.

> Set your identity first if Git asks:
> ```bash
> git config --global user.name "Your Name"
> git config --global user.email "you@example.com"
> ```

### 11.2 Create an empty remote repository

On **GitHub** (or GitLab/Bitbucket), create a **new empty repository** — do **not** add a README, .gitignore, or license (you already have them locally). Name it e.g. `bookish`. Copy the repo URL it shows you.

### 11.3 Connect and push

Use **one** of the two auth methods.

**Option A — HTTPS (uses a Personal Access Token as the password):**

```bash
git remote add origin https://github.com/YOUR_USERNAME/bookish.git
git branch -M main
git push -u origin main
```

When prompted for a password, paste a **Personal Access Token** (GitHub → Settings → Developer settings → Personal access tokens → generate one with `repo` scope). Your normal account password will not work for pushing.

**Option B — SSH (no password prompt once a key is set up):**

```bash
# One-time: create a key if you don't have one, then add the
# contents of ~/.ssh/id_ed25519.pub to GitHub → Settings → SSH keys.
ssh-keygen -t ed25519 -C "you@example.com"

git remote add origin git@github.com:YOUR_USERNAME/bookish.git
git branch -M main
git push -u origin main
```

### 11.4 Day-to-day after the first push

```bash
git add .
git commit -m "Describe what you changed"
git push
```

That's it — your code is now on the remote and `git push` syncs future changes.

> **Never commit `.env`.** It holds your database credentials and API tokens. The `.gitignore` already protects you; keep it that way. When deploying to a server, set the environment variables there separately.

## 12. Notes & next steps

- The Blade views use the **Tailwind CDN** so they render with zero build step. For production you'd typically install Tailwind via npm and compile, but it's not required to run.
- Plug in live **WhatsApp Cloud API** and **SMS gateway** credentials in `.env` to send real OTPs.
- Update the store **bank details** in the admin Settings module (super-admin) so the checkout page shows your real account / Raast ID.
- Change the **default super-admin password** right after first login.
- "Baby garments" is noted in the spec as future expansion — the category system is dynamic, so you can add it from the admin panel without code changes.
